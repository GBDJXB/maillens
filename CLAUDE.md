# MailLens — Project Guide for Claude Code

This file tells Claude Code how to work in this repository. Read it fully before making changes. When in doubt about a design decision, check the **Key Decisions & Rationale** section at the bottom — many choices here are deliberate and should not be "fixed."

---

## 1. Project Overview

**MailLens** is a local-first email search engine. Users import their historical emails (from IMAP accounts, `.mbox` files, or Outlook `.pst` files) into a local database, then search them using a hybrid approach combining BM25 keyword search and semantic vector search.

This is a **personal learning project** whose primary purpose is to demonstrate engineering competence on a GitHub portfolio. Secondary purpose is real personal use. It is not a commercial product.

The user is building this to learn modern AI/data engineering tooling hands-on. Code quality, clear architecture, and thoughtful documentation matter as much as functionality.

---

## 2. Goals

- **Local-first.** All data stays on the user's machine. No cloud dependencies for core functionality.
- **Low-friction import.** Users must be able to import historical emails with minimal technical steps. IMAP with auto-detected host based on email domain is the primary path.
- **Hybrid search quality.** Combine BM25 (via SQLite FTS5) and dense vector search (via LanceDB) with Reciprocal Rank Fusion and cross-encoder reranking.
- **Cross-platform stability with Windows priority.** The software must run reliably on Windows first, macOS second, Linux third. Any platform-specific code path must default to the Windows-safe version.
- **Library-first architecture.** Core logic lives in an importable library (`maillens.core`). The CLI is a thin wrapper. Future AI/agent features will be additional consumers of the same core.
- **Evaluation-driven.** Search quality is measured with a real eval harness (recall@10, MRR) on a labeled test set.

---

## 3. Non-Goals (MVP)

Do not add scope in any of these directions without explicit user instruction:

- **No LLM agent in MVP.** Query parsing uses rules/regex. Agent/tool-use architecture is explicitly deferred to a future version. The core library must expose clean APIs so an agent layer can be added later.
- **No GUI in MVP.** CLI only. GUI (if ever) is a v2 concern.
- **No cloud sync, no multi-user, no server deployment.**
- **No attachment content indexing.** Attachment metadata only (filename, content-type, size).
- **No WeChat or any non-standard Chinese messaging platforms.** These have no official API and create legal risk.
- **No scraping of Gmail/Outlook via unofficial means.** Use only IMAP or official APIs.
- **No microservices, no Docker in MVP, no background daemon.** It is a local CLI tool.

---

## 4. Platform Priority and Implications

**Windows > macOS > Linux.** Concretely this means:

- Primary dev/test target is Windows. Claude Code should assume the user runs Windows and write code that works there first.
- Any file system operation uses `pathlib.Path`. Never concatenate paths with string operators or assume forward slashes.
- File opens always pass `encoding="utf-8"` explicitly. Windows default encoding (cp1252/GBK) will break things.
- SQLite database paths must be converted to forward-slash form via `Path.as_posix()` before being embedded in a SQLAlchemy URL.
- Do not rely on POSIX-only features: `os.fork`, symlinks that require admin, signals other than SIGINT, case-insensitive path resolution, or `/tmp` being writable without `tempfile`.
- Subprocess calls must pass `shell=False` and use list arguments; do not depend on `/bin/sh` or `which`.
- Long path support: file names generated by the app should stay well under 200 characters.

## 6. Tools and Libraries to Avoid

- **LangChain, LlamaIndex.** Too much abstraction, hides the actual control flow, reduces learning value, and adds version-churn risk. Use raw LLM APIs + typed tool-calling when LLM features arrive.
- **Chroma.** Runs as a background process by default; conflicts with our "single-file, no daemon" goal. Use LanceDB.
- **SQLAlchemy ORM layer.** Overkill for our schema. Use Core.
- **`pydantic.EmailStr`.** Real-world emails contain addresses that fail strict validation (bot senders, malformed legacy addresses). Use plain `str` for email fields.
- **PyQt, tkinter, wxPython.** Packaging pain on Windows. If GUI becomes needed, prefer `pywebview` or Tauri with a Python backend.
- **`libpff` / `pypff` Python bindings.** Windows compilation issues. Shell out to external `readpst` instead.
- **`requirements.txt`.** Use `pyproject.toml` + `uv.lock` instead.
- **`setup.py`.** Use `pyproject.toml` exclusively.

## 8. Coding Conventions

- **Type hints everywhere.** Public functions must have complete annotations. `mypy --strict` should pass on `core/`.
- **Pydantic v2 for all data that crosses module boundaries.** Internal helpers can use plain dataclasses or tuples.
- **No mutable default arguments.** Use `Field(default_factory=list)` in pydantic, `field(default_factory=list)` in dataclasses.
- **Explicit UTF-8.** Every `open()` passes `encoding="utf-8"`. Every decode passes `errors="replace"` unless correctness demands failure.
- **Paths are `pathlib.Path`.** Conversion to str only at the boundary with libraries that need it.
- **No bare `except:`.** Catch specific exceptions. If you must catch broad exceptions (e.g. in an ingester loop), log with the exception type and continue.
- **Logging, not printing, inside `core/`.** Use `logging.getLogger(__name__)`. CLI layer configures log handlers.
- **Small functions.** If a function exceeds ~40 lines, consider splitting.
- **Docstrings on public APIs.** Google or NumPy style, pick one (Google recommended) and stick to it.
- **Commit messages follow Conventional Commits** (`feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`).

## 9. Data Model Essentials

Canonical `Message` model lives in `src/maillens/core/models.py`. Key fields and reasoning:

- `message_id: str` — RFC 5322 Message-ID if present, otherwise synthetic (`<synthetic-{uuid}@maillens>`). Used as primary key.
- `source: str` — one of `"mbox"`, `"imap"`, `"pst"`, etc. For provenance.
- `source_ref: str | None` — source-specific identifier (e.g., IMAP UID) for later re-fetching.
- `thread_id: str | None` — computed by MailLens, not from the email itself. Based on `References` + `In-Reply-To` headers.
- `sent_at: datetime` — parsed from `Date` header, timezone-aware.
- `sender: Contact` — `email: str` plus optional `name: str`. Email is **not** validated as EmailStr.
- `recipients: list[tuple[Contact, RecipientKind]]` — single flat list, not separate to/cc/bcc lists. Makes iteration natural.
- `body_text: str` — cleaned plain text. HTML is stripped before storage.
- `body_html: str | None` — kept optionally for display.
- `attachments: list[Attachment]` — metadata only in MVP (filename, content_type, size_bytes).

Database schema must mirror this model but denormalizes contacts into a separate `contacts` table with a junction table for recipients.

## 10. Ingestion Pipeline Expectations

All ingesters implement:

```python
class Ingester(ABC):
    @abstractmethod
    def iter_messages(self) -> Iterator[Message]: ...
```

The iterator must be **tolerant**: parsing a single malformed message should not abort the run. Catch the exception, log it with enough context to find the offending message, and continue.

**IMAP ingester specifics:**
- Auto-detect host from email domain via a lookup table (Gmail, Outlook.com, Yahoo, QQ, 163, 126 minimum).
- Always use IMAPS (port 993), never plain IMAP.
- Accept app-specific passwords. Document in the README how to generate one for each major provider.
- Support resumption: record last processed UID per folder in the database; on next run, fetch only UIDs greater than that.

**mbox ingester specifics:**
- Handle MIME-encoded headers via `email.header.decode_header` + `make_header`.
- Prefer `text/plain` parts; fall back to HTML stripped to text.
- Use `get_content_charset()` with `errors="replace"` on decode.
- **Date parsing fallback chain (do not simplify this):** Try `Date` header first. If missing or unparseable, try the timestamp embedded in the bottom-most `Received` header (after the last `;`). If that also fails, set `sent_at` to the epoch sentinel `datetime(1970, 1, 1, tzinfo=timezone.utc)`. Never use `datetime.now()` as a fallback — it silently lies about when the email was sent. The epoch value makes broken dates visibly wrong when sorting.

**PST ingester specifics (Week 2+):**
- Shell out to `readpst -e -o <outdir> <pstfile>` to convert PST to mbox files per folder.
- Then reuse the mbox ingester on the output.
- Ship a clear error message if `readpst` is not on PATH, with install instructions per platform.

## 11. Search Architecture

**Two-index design:**

1. SQLite + FTS5 for metadata and keyword search. FTS5 virtual table indexes `subject` and `body_text` with `tokenize='unicode61'`.
2. LanceDB for dense vectors. Each row stores `(message_id, chunk_index, vector, text_snippet)`.

**Chunking rules:**
- Messages with `body_text` under ~500 tokens: index whole message as one chunk.
- Longer messages: split by paragraph, each chunk prefixed with `[Subject: ...] [From: ...] [Date: ...]` context.
- Also index the full thread (concatenated messages) as its own document for thread-level queries.

**Query flow:**
1. Parse query into `{semantic_query, filters, required_terms}` using regex-based rules. Examples of filter syntax: `from:alice`, `after:2024-01-01`, `folder:Sent`. Everything else is semantic.
2. Run FTS5 keyword search and LanceDB vector search in parallel.
3. Merge with Reciprocal Rank Fusion (`k=60`).
4. Apply structured filters.
5. Rerank top-50 with cross-encoder, return top-10.
6. Deduplicate by `message_id` (multiple chunks from the same message collapse to one result).

## 12. Evaluation Framework

Located in `src/maillens/core/eval/`. The user will hand-label ~50 queries on the Enron dataset (or a subset of their own mail) with expected message IDs.

Required metrics:
- Recall@10
- MRR (Mean Reciprocal Rank)

Required ablations (run as part of `maillens eval` command):
- FTS only
- Vector only
- Hybrid without rerank
- Hybrid with rerank

Results must be serializable to a Markdown table for the README.

## 17. Privacy and Security

- The default embedding model is local (`all-MiniLM-L6-v2`). Any cloud embedding option (OpenAI, Voyage) must be opt-in via explicit config, never default.
- Credentials (IMAP passwords) are stored in the OS keychain via the `keyring` library, never in plaintext config files.
- The database file must have 0600 permissions (owner read/write only) on POSIX systems.
- Logs must never include message bodies, subjects, or credentials. Message IDs and counts are fine.
- When a user runs `maillens reset`, all local data is deleted cleanly (database, vector store, keyring entries).

## 18. Key Decisions & Rationale

Documented here so these choices are not accidentally reversed.

### Why SQLite instead of Postgres
This is a single-user local tool. SQLite is a file; Postgres needs a running service. SQLite handles hundreds of thousands of emails easily with FTS5.

### Why LanceDB instead of Chroma or FAISS
LanceDB is embedded (no background process), file-based (easy backup), and has a clean pandas-like API. Chroma defaults to client-server mode. FAISS has no persistence layer and no metadata support — we'd be reinventing LanceDB.

### Why SQLAlchemy Core instead of ORM
The schema is simple enough that ORM is overkill. Core keeps SQL visible, which is pedagogically better and performs better for bulk inserts.

### Why `all-MiniLM-L6-v2` as the default embedding model
Smallest mainstream model (80 MB), CPU-runnable, well-documented, supported by every vector database. Quality is good enough for English email. Larger models (bge-m3, nomic-embed) can be swapped in via config.

### Why no LangChain / LlamaIndex
Control flow transparency matters for a learning project. Framework abstractions hide the very mechanics the user wants to learn. Direct API calls + typed tool-calling are clearer and more durable.

### Why Windows-first despite Python's Linux bias
The user's primary target audience is Windows users. Developing Windows-first catches compatibility issues early rather than in a panicked Week 9.

### Why CLI-only in MVP
Scope control. A GUI doubles the surface area of testing, packaging, and cross-platform work without adding learning value for backend/AI engineering roles.

### Why `readpst` instead of `libpff` Python bindings
`libpff` requires C compilation that is fragile on Windows. `readpst` is a stable binary available via the `libpst` package on every major platform. Shelling out is less elegant but dramatically more reliable.

### Why pydantic email field is plain `str`, not `EmailStr`
Real-world email archives contain addresses that fail RFC validation (bot senders, legacy malformed addresses). Strict validation would drop legitimate data.

### Why library-first (core/cli separation)
Future AI features (agent, natural-language query, summarization) should be additional consumers of the same core library. Baking logic into the CLI would force a rewrite later.

### Why evaluate with recall@10 and MRR
Standard metrics in information retrieval research. Comparable to published benchmarks. MRR specifically measures whether the first hit is good, which matches how users actually use search.
